Visitor Management Guide

Comprehensive visitor management with pre-approvals, check-ins, and real-time notifications.

Overview

GateFlux provides comprehensive visitor management capabilities, allowing residents to pre-approve visitors, security to manage check-ins, and administrators to monitor visitor activity across the society.

Visitor Types

Type Description
GuestPersonal visitors to residents
DeliveryE-commerce, food delivery, courier
CabTaxi, rideshare services
ServiceRepair, maintenance, utility workers
VendorRegular service providers
EmergencyMedical, fire, police

Pre-Approving Visitors

Resident Pre-Approval (Mobile App)

Residents can pre-approve expected visitors through the mobile app:

  1. Open GateFlux mobile app
  2. Navigate to Visitors → Add Visitor
  3. Enter visitor details
  4. Set expected arrival time
  5. Submit for approval

Pre-Approval via API

HTTP 
POST /api/flats/{flat_id}/visitors/pre-approve
Authorization: Bearer <resident_token>
Content-Type: application/json

{
    "name": "Visitor Name",
    "phone": "+91-9876543210",
    "purpose": "Personal Visit",
    "visitor_type": "guest",
    "expected_arrival": "2024-02-16T14:00:00",
    "expected_departure": "2024-02-16T18:00:00",
    "vehicle_number": "MH01AB1234",
    "vehicle_type": "car",
    "notes": "Coming for birthday party"
}

Response

JSON 
{
    "success": true,
    "data": {
        "id": 12345,
        "approval_code": "VIS-2024-ABC123",
        "qr_code": "<base64_encoded_qr>",
        "status": "pre_approved",
        "valid_until": "2024-02-16T23:59:59"
    }
}

Gate Check-In Process

Security Guard Check-In

When a visitor arrives at the gate:

Step 1: Verify Pre-Approval

HTTP 
GET /api/visitors/verify?code=VIS-2024-ABC123
Authorization: Bearer <guard_token>

Step 2: Check-In Visitor

HTTP 
POST /api/visitors/{visitor_id}/check-in
Authorization: Bearer <guard_token>
Content-Type: application/json

{
    "id_proof_type": "driving_license",
    "id_proof_number": "MH-1234567890",
    "photo": "<base64_encoded_photo>",
    "vehicle_number": "MH01AB1234",
    "entry_gate": "Main Gate"
}

Walk-In Visitor (No Pre-Approval)

For visitors without pre-approval:

HTTP 
POST /api/visitors/walk-in
Authorization: Bearer <guard_token>
Content-Type: application/json

{
    "name": "Walk-in Visitor",
    "phone": "+91-9876543210",
    "flat_id": 101,
    "purpose": "Meeting",
    "visitor_type": "guest",
    "id_proof_type": "aadhaar",
    "id_proof_number": "1234-5678-9012",
    "photo": "<base64_encoded_photo>",
    "vehicle_number": "MH01AB1234"
}
Note

This triggers a notification to the resident for approval.

Resident Approval for Walk-In

HTTP 
POST /api/visitors/{visitor_id}/approve
Authorization: Bearer <resident_token>
Content-Type: application/json

{
    "action": "approve",
    "notes": "Please send them up"
}

Or reject:

JSON 
{
    "action": "reject",
    "reason": "Not expecting any visitors"
}

Visitor Check-Out

HTTP

{

    "notes": "Departed with belongings"

Delivery Management

Handling Deliveries

HTTP 
POST /api/deliveries
Authorization: Bearer <guard_token>
Content-Type: application/json

{
    "flat_id": 101,
    "delivery_type": "food",
    "delivery_company": "Swiggy",
    "delivery_person_name": "Delivery Agent",
    "delivery_person_phone": "+91-9876543210",
    "package_description": "Food order",
    "allow_entry": false
}

Delivery Options

Option Description
allow_entry = true Delivery person goes to flat
allow_entry = false Resident collects from gate
leave_at_gate Package left at security

Visitor History & Reports

View Visitor History

HTTP 
GET /api/societies/{society_id}/visitors?from=2024-01-01&to=2024-02-16
Authorization: Bearer <admin_token>

Filter Options

Parameter Description
fromStart date (YYYY-MM-DD)
toEnd date (YYYY-MM-DD)
flat_idFilter by specific flat
visitor_typeFilter by type (guest, delivery, etc.)
statusFilter by status (checked_in, checked_out)

Export Visitor Report

HTTP 
GET /api/societies/{society_id}/visitors/export?format=csv&from=2024-01-01&to=2024-02-16
Authorization: Bearer <admin_token>

Frequent Visitor Management

Add Frequent Visitor

For regular visitors like tutors, nurses, etc.:

HTTP 
POST /api/flats/{flat_id}/frequent-visitors
Authorization: Bearer <resident_token>
Content-Type: application/json

{
    "name": "Frequent Visitor Name",
    "phone": "+91-9876543210",
    "relation": "Tutor",
    "valid_from": "2024-01-01",
    "valid_until": "2024-12-31",
    "allowed_days": ["monday", "wednesday", "friday"],
    "allowed_time_from": "16:00",
    "allowed_time_to": "18:00",
    "photo": "<base64_encoded_photo>"
}
Tip

Frequent visitors can enter without per-visit approval during allowed times.

Blacklisted Visitors

Add to Blacklist

HTTP 
POST /api/societies/{society_id}/blacklist
Authorization: Bearer <admin_token>
Content-Type: application/json

{
    "name": "Person Name",
    "phone": "+91-9876543210",
    "id_proof_number": "1234-5678-9012",
    "reason": "Previous misconduct",
    "photo": "<base64_encoded_photo>"
}
Warning

Blacklisted individuals will trigger alerts when attempting entry.

Notifications

Visitor Notifications

GateFlux sends notifications for:

Event Recipients
Visitor at gateResident
Visitor approvedSecurity guard
Visitor rejectedSecurity guard
Visitor checked inResident
Visitor checked outResident (optional)
Suspicious activityAdmin, Security

Configure Notification Preferences

HTTP 
PUT /api/users/{user_id}/notification-preferences
Authorization: Bearer <token>
Content-Type: application/json

{
    "visitor_arrival": true,
    "visitor_departure": false,
    "delivery_arrival": true,
    "push_notifications": true,
    "sms_notifications": false
}

Security Best Practices

Best Practices for Security
  1. Always verify ID proof for first-time visitors
  2. Capture photos of all visitors
  3. Record vehicle numbers for parking management
  4. Use OTP verification for cab services
  5. Report suspicious activity immediately
  6. Regular blacklist review to keep it updated
