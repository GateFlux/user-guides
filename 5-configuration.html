Configuration Guide

Configure GateFlux for your society's specific requirements including environment setup, system settings, and customization options.

Environment Configuration

API Configuration (.env)

The API configuration file is located at api/.env. Key settings:

ENV 
# Application Settings
APP_NAME=GateFlux
APP_ENV=production
APP_KEY=base64:your_generated_key
APP_DEBUG=false
APP_URL=https://api.yourdomain.com

# Database Configuration
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=gateflux
DB_USERNAME=gateflux_user
DB_PASSWORD=secure_password

# Cache & Session
CACHE_DRIVER=redis
SESSION_DRIVER=redis
QUEUE_CONNECTION=redis

# Redis Configuration
REDIS_HOST=127.0.0.1
REDIS_PASSWORD=null
REDIS_PORT=6379

# Mail Configuration
MAIL_MAILER=smtp
MAIL_HOST=smtp.mailgun.org
MAIL_PORT=587
MAIL_USERNAME=your_username
MAIL_PASSWORD=your_password
MAIL_ENCRYPTION=tls
MAIL_FROM_ADDRESS=noreply@yourdomain.com
MAIL_FROM_NAME="${APP_NAME}"

# SMS Configuration
SMS_PROVIDER=twilio
TWILIO_SID=your_twilio_sid
TWILIO_TOKEN=your_twilio_token
TWILIO_FROM=+1234567890

# Push Notifications
FIREBASE_PROJECT_ID=your_project_id
FIREBASE_CREDENTIALS=/path/to/firebase-credentials.json

# File Storage
FILESYSTEM_DISK=s3
AWS_ACCESS_KEY_ID=your_access_key
AWS_SECRET_ACCESS_KEY=your_secret_key
AWS_DEFAULT_REGION=ap-south-1
AWS_BUCKET=gateflux-uploads

Backup Configuration

Database Backup

Configure automated backups in api/config/backup.php:

PHP 
return [
    'backup' => [
        'name' => 'gateflux-backup',
        'source' => [
            'databases' => ['mysql'],
        ],
        'destination' => [
            'disks' => ['s3'],
        ],
    ],
    'cleanup' => [
        'strategy' => 'default',
        'default' => [
            'keep_all_backups_for_days' => 7,
            'keep_daily_backups_for_days' => 30,
            'keep_weekly_backups_for_weeks' => 12,
        ],
    ],
];

Schedule Backups

Add to api/app/Console/Kernel.php:

PHP 
$schedule->command('backup:run')->dailyAt('02:00');
$schedule->command('backup:clean')->dailyAt('03:00');

Registration Settings

GateFlux supports two types of self-service registration:

User Self-Signup

When allow_self_signup is enabled per-tenant, residents can register themselves on their society's login page:

  • Users select their society from a dropdown
  • New users require approval from society admins
  • Enable via Tenant Management: set allow_self_signup = true on the tenant

Society/Tenant Registration

When allow_tenant_registration is enabled, new societies can request to join the platform:

HTTP 
PUT /api/v1/platform/settings
Authorization: Bearer <super_admin_token>
Content-Type: application/json

{
    "allow_tenant_registration": true
}

When enabled, a "Register Society" link appears on the login page. Registration requests are submitted as demo requests for Super Admin review.

Two-Factor Authentication (2FA)

GateFlux supports TOTP-based Two-Factor Authentication using apps like Google Authenticator, Authy, or Microsoft Authenticator.

Enabling 2FA for All Admins

Super Admins can require 2FA for all admin users platform-wide:

HTTP 
PUT /api/v1/platform/settings
Authorization: Bearer <super_admin_token>
Content-Type: application/json

{
    "two_factor_required": true
}

When enabled, builder_admin and super_admin users must set up 2FA before accessing the platform.

User 2FA Setup

HTTP 
# 1. Get 2FA status
GET /api/v1/auth/2fa/status

# 2. Start setup (generates QR code)
POST /api/v1/auth/2fa/setup

Response:
{
    "data": {
        "secret": "JBSWY3DPEHPK3PXP...",
        "qr_code": "data:image/svg+xml;base64,...",
        "manual_entry_key": "JBSW Y3DP EHPK 3PXP ..."
    }
}

# 3. Confirm with 6-digit code from authenticator
POST /api/v1/auth/2fa/confirm
{
    "code": "123456"
}

Response includes recovery codes - save these securely!

2FA During Login

When a user with 2FA enabled logs in, the login response will indicate 2FA is required:

HTTP 
# Login returns requires_2fa instead of token
POST /api/v1/auth/login
{
    "email": "admin@tenant.com",
    "password": "password"
}

Response:
{
    "data": {
        "requires_2fa": true,
        "user_id": "uuid-here",
        "user": { "name": "Admin", "email": "admin@tenant.com" }
    }
}

# Then verify with 2FA code
POST /api/v1/auth/2fa/verify
{
    "user_id": "uuid-here",
    "code": "123456"
}

# Or use a recovery code
POST /api/v1/auth/2fa/verify
{
    "user_id": "uuid-here",
    "recovery_code": "ABCD-1234"
}

Disabling 2FA

HTTP 
POST /api/v1/auth/2fa/disable
{
    "password": "current_password",
    "code": "123456"
}

Maintenance Mode

Enable maintenance mode to block tenant access during updates:

HTTP 
PUT /api/v1/platform/settings
Authorization: Bearer <super_admin_token>
Content-Type: application/json

{
    "maintenance_mode": true
}

When enabled, all tenant API endpoints return 503 Service Unavailable. Super Admins can still access the platform.

Queue Workers

Configure queue workers in supervisor.conf:

INI 
[program:gateflux-worker]
process_name=%(program_name)s_%(process_num)02d
command=php /var/www/gateflux/api/artisan queue:work redis --sleep=3 --tries=3 --max-time=3600
autostart=true
autorestart=true
stopasgroup=true
killasgroup=true
numprocs=4
redirect_stderr=true
stdout_logfile=/var/log/gateflux-worker.log
